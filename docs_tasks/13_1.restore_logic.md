# 13_1. restore logic

## 0. 목적
- retention 정책으로 남겨둔 과거 버전 인덱스 목록을 조회할 수 있어야 한다.
- 운영자가 그중 하나를 선택해서 현재 `readAlias`를 해당 인덱스로 되돌릴 수 있어야 한다.
- 목표는 복잡한 복구 시스템이 아니라, **alias 재전환 기반의 빠른 rollback 절차**를 만드는 것이다.

## 1. 목표 작업
1. 현재 `indexName` prefix에 해당하는 버전 인덱스 목록 조회
2. 현재 `readAlias`가 연결된 인덱스 식별
3. rollback 가능한 후보 인덱스 목록 정리
4. 선택된 인덱스로 `readAlias` 재전환
5. 전환 후 검색 검증

## 2. 조회 대상 기준
- 대상 인덱스:
  - `${ai-search.index-name}-*`
- 예:
  - `food-products-v20260316103000`
  - `food-products-v20260316110000`
  - `food-products-v20260316113000`
- 제외 대상:
  - 시스템 인덱스
  - alias 이름 자체

## 3. 과거 인덱스 조회 작업

### 3.1 필요한 정보
- `indexName`
  - 예: `food-products`
- `readAlias`
  - 예: `food-products-read`

### 3.2 조회 절차
1. `${indexName}-*` 패턴으로 인덱스 목록 조회
2. 인덱스명을 문자열 내림차순 정렬
3. 현재 `readAlias`가 가리키는 인덱스 조회
4. 결과에 아래 정보 포함
  - indexName
  - 현재 alias 연결 여부
  - rollback 후보 여부

### 3.3 기대 결과 예시
| index | alias 연결 여부 | rollback 후보 |
|---|---|---|
| `food-products-v20260316113000` | Y | N |
| `food-products-v20260316110000` | N | Y |
| `food-products-v20260316103000` | N | Y |

## 4. restore 실행 작업

### 4.1 입력값
- `targetIndex`
  - 운영자가 선택한 과거 버전 인덱스

### 4.2 실행 절차
1. `targetIndex` 존재 여부 확인
2. `targetIndex`가 `${indexName}-*` 규칙에 맞는지 확인
3. 현재 `readAlias` 대상 인덱스 조회
4. 현재 인덱스와 `targetIndex`가 같으면 작업 종료
5. alias를 현재 인덱스에서 제거
6. alias를 `targetIndex`에 연결
7. 전환 결과 재조회

### 4.3 안전 조건
- `targetIndex`가 존재하지 않으면 실패
- `targetIndex`가 현재 alias 대상과 같으면 `400 Bad Request`
- `readAlias`는 항상 정확히 1개 인덱스만 가리키게 유지

## 5. 예상 코드 변경 범위
- `AliasSwitcher.java`
  - 현재 alias 대상 조회 메서드 재사용 또는 확장
  - 특정 target index로 alias 전환 메서드 추가
- `IndexCleanupService.java`
  - retention 정책과 충돌하지 않도록 rollback 대상 인덱스 보호
- 신규 서비스 예시:
  - `IndexRestoreService.java`
    - 과거 인덱스 목록 조회
    - target index restore 실행
- 신규 DTO 예시:
  - `RestoreIndexCandidate`
  - `RestoreIndexResult`

## 5_1. 권장 구현 방식
- 가장 현실적인 구현은 **REST API 2개 추가 방식**이다.
- 이유:
  - 운영자가 ES 인덱스명을 직접 조회하지 않아도 된다.
  - UI, 운영 스크립트, 수동 API 호출에서 공통으로 재사용 가능하다.
  - 현재 alias 전환 로직을 그대로 재사용할 수 있다.
  - 추후 인증/인가, 승인 절차, 감사 로그를 붙이기 쉽다.

### 5_1.1 REST API Spec: 복구 가능한 인덱스 목록 조회

**Method**
- `GET`

**Path**
- `/api/admin/index-restore/candidates`

**Description**
- 현재 `readAlias` 기준으로 복구 가능한 버전 인덱스 목록을 조회한다.
- `${indexName}-v*` 패턴 인덱스만 조회 대상이다.
- 현재 alias가 가리키는 인덱스도 목록에 포함하되, `restorable=false`로 반환한다.

**Request**
- Query Parameter 없음
- Request Body 없음

**Response 200**
```json
{
  "alias": "food-products-read",
  "currentAliasIndex": "food-products-v20260316113000",
  "retentionCount": 3,
  "count": 3,
  "candidates": [
    {
      "indexName": "food-products-v20260316113000",
      "current": true,
      "restorable": false
    },
    {
      "indexName": "food-products-v20260316110000",
      "current": false,
      "restorable": true
    },
    {
      "indexName": "food-products-v20260316103000",
      "current": false,
      "restorable": true
    }
  ]
}
```

**Response Field**
| 필드 | 타입 | 설명 |
|---|---|---|
| `alias` | `string` | 현재 운영 검색 alias |
| `currentAliasIndex` | `string \| null` | 현재 alias가 가리키는 물리 인덱스 |
| `retentionCount` | `number` | 현재 retention 설정값 |
| `count` | `number` | 반환된 후보 인덱스 개수 |
| `candidates[].indexName` | `string` | 버전 인덱스명 |
| `candidates[].current` | `boolean` | 현재 alias 대상 여부 |
| `candidates[].restorable` | `boolean` | restore 가능 여부 |

**정렬 규칙**
- `indexName` 문자열 내림차순 정렬
- 최신 인덱스가 먼저 반환된다

**Response 500**
- alias 조회 실패
- 인덱스 목록 조회 실패

### 5_1.2 REST API Spec: 선택 인덱스로 restore 실행

**Method**
- `POST`

**Path**
- `/api/admin/index-restore`

**Description**
- 운영자가 지정한 과거 버전 인덱스로 `readAlias`를 되돌린다.
- 실제 데이터 재색인은 수행하지 않고, alias 전환만 수행한다.

**Request Body**
```json
{
  "targetIndex": "food-products-v20260316103000"
}
```

**Request Field**
| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `targetIndex` | `string` | Y | restore 대상 버전 인덱스명 |

**검증 규칙**
- `targetIndex`는 비어 있으면 안 된다
- `targetIndex`는 `${indexName}-v*` 패턴이어야 한다
- `targetIndex`는 실제 존재해야 한다
- `targetIndex`가 현재 alias 대상 인덱스와 같으면 `400 Bad Request`

**Response 200**
```json
{
  "success": true,
  "alias": "food-products-read",
  "oldIndex": "food-products-v20260316113000",
  "restoredIndex": "food-products-v20260316103000"
}
```

**Response Field**
| 필드 | 타입 | 설명 |
|---|---|---|
| `success` | `boolean` | restore 성공 여부 |
| `alias` | `string` | 전환된 alias 이름 |
| `oldIndex` | `string \| null` | restore 직전 alias 대상 인덱스 |
| `restoredIndex` | `string` | restore 후 alias가 가리키는 인덱스 |

**Response 400**
- `targetIndex`가 비어 있음
- `targetIndex` 형식이 잘못됨
- `targetIndex`가 현재 alias 대상과 동일함

**Response 404**
- `targetIndex`가 존재하지 않음

**Response 500**
- alias 전환 실패
- restore 결과 검증 실패

### 5_1.3 내부 구현 방향
- `IndexRestoreService`
  - `listCandidates()`
  - `restoreTo(String targetIndex)`
- 자동 테스트용 롤아웃 경로 확장
  - `FoodDataLoader.loadAll()` 유지
  - `FoodDataLoader.loadAll(String path)` 오버로딩 추가
  - `ProductIndexingService.reindexData(String indexName, String dataPath)` 오버로딩 추가
  - `IndexRolloutService.rollOutFromSourceData(String dataPath)` 오버로딩 추가
- `AliasSwitcher` 재사용
  - 현재 alias 대상 조회
  - target index로 alias 전환
- `${indexName}-v*`만 restore 후보로 취급
- 인덱스명 문자열 내림차순 정렬
- 현재 alias 대상 인덱스는 목록에 포함하되 `restorable=false`
- `targetIndex`가 현재 alias 대상이면 `400 Bad Request`

## 5_3. 구현 시 주의사항
- 운영 API이므로 `/api/admin/...` 경로로 분리하는 것이 좋다.
- 현재 프로젝트 보안 상태를 고려하면 상용 전에는 인증/인가를 반드시 붙여야 한다.
- restore API는 실제 데이터 재색인이 아니라 **alias 전환만 수행**해야 한다.
- restore 대상은 반드시 `${indexName}-v*` 패턴과 실제 존재 여부를 함께 검증해야 한다.
- alias는 항상 단일 인덱스만 가리키도록 유지해야 한다.

## 8. 테스트 케이스 기반 자동 검증 시나리오

### 8.1 목표
- Spring Boot 테스트 코드에서 웹서버를 띄운 상태로 restore 기능을 end-to-end 검증한다.
- 검증 흐름은 다음과 같다.
  1. 정상 원본 데이터로 baseline 인덱스를 만든다.
  2. 10건 전용 데이터로 신규 인덱스를 만든다.
  3. 검색 결과 이상을 확인한다.
  4. restore 후보 목록 API를 조회한다.
  5. 직전 인덱스로 rollback 한다.
  6. 검색 결과가 정상으로 복구됐는지 확인한다.

### 8.2 테스트 데이터
- 기준 원본 파일:
  - `src/main/resources/data/food-products.json`
- 10건 전용 테스트 파일:
  - `src/main/resources/data/food-products_only10.json`
- 검색 이상 재현용 키워드:
  - `치즈`

이유:
- `food-products_only10.json`의 10개 상품에는 `치즈` 관련 상품이 없다.
- 원본 `food-products.json`에는 `올가 유기농 슬라이스 치즈` 상품이 존재한다.
- 따라서 rollback 전후 검색 결과 차이를 자동 검증하기 쉽다.

### 8.3 자동 테스트 시나리오
1. `FoodDataLoader.loadAll()`은 기본 경로 `data/food-products.json`를 사용한다.
2. `FoodDataLoader.loadAll(String path)` 오버로딩을 추가한다.
3. `ProductIndexingService.reindexData(String indexName, String dataPath)` 오버로딩을 추가한다.
4. `IndexRolloutService.rollOutFromSourceData(String dataPath)` 오버로딩을 추가한다.
5. baseline 롤아웃은 기본 `rollOutFromSourceData()`로 수행한다.
6. `q=치즈` 검색 결과에 치즈 상품이 존재하는지 확인한다.
7. 10건 전용 롤아웃은 `rollOutFromSourceData("data/food-products_only10.json")`로 수행한다.
8. `q=치즈` 검색 결과가 비어 있거나 기대 상품이 없음을 확인한다.
9. `GET /api/admin/index-restore/candidates` 호출
10. restore 가능한 인덱스가 1개 이상 존재하는지 확인한다.
11. restore 가능한 인덱스 중 가장 최근 인덱스를 선택한다.
12. `POST /api/admin/index-restore`에 `targetIndex`를 넣어 rollback 한다.
13. 다시 `q=치즈` 검색을 수행한다.
14. 치즈 상품이 다시 검색되는지 확인한다.

### 8.4 테스트 구현 권장 방식
- 테스트 종류:
  - `@SpringBootTest(webEnvironment = RANDOM_PORT)`
- 호출 방식:
  - 실제 HTTP 호출
  - `/api/search`
  - `/api/admin/index-restore/candidates`
  - `/api/admin/index-restore`
- 테스트 클래스 예시:
  - `src/test/java/com/example/aisearch/integration/indexing/IndexRestoreIntegrationTest.java`

### 8.5 테스트 코드에서 필요한 보조 작업
- baseline 인덱스 생성용 롤아웃 호출
- 10건 전용 경로를 사용하는 롤아웃 호출
- 최신 인덱스명 / restore 후보 목록 파싱
- 직전 인덱스 선택 로직

### 8.5_1 권장 로더 구조
- `loadAll()`
  - 기존 기본 동작 유지
  - 내부적으로 `data/food-products.json` 사용
- `loadAll(String path)`
  - 테스트 또는 특수 롤아웃에서 명시한 경로 사용
- `reindexData(String indexName, String dataPath)`
  - 지정한 데이터 경로로만 색인 수행
- `rollOutFromSourceData(String dataPath)`
  - 롤아웃 전체 흐름에서 테스트 데이터 경로를 전달
- 이 방식이면 테스트가 실제 리소스 파일을 덮어쓰지 않아도 된다.

### 8.6 기대 검증 포인트
- baseline 상태에서는 `치즈` 검색 결과가 정상이다.
- 10건 인덱스 상태에서는 `치즈` 검색 결과가 비정상이다.
- restore 후보 목록에 직전 인덱스가 존재한다.
- rollback 후 `치즈` 검색 결과가 다시 정상이다.
- alias는 rollback 후 선택한 과거 인덱스를 가리킨다.

### 8.7 구현 전 추가 확인 필요 사항
- 현재 `FoodDataLoader`는 `food-products.json`를 고정 경로로 읽는다.
- 자동 테스트를 안정적으로 만들기 위해 이번 작업에서는 아래 방식으로 정리한다.
  1. `loadAll()` 유지
  2. `loadAll(String path)` 오버로딩 추가
  3. `reindexData(String indexName, String dataPath)` 오버로딩 추가
  4. `rollOutFromSourceData(String dataPath)` 오버로딩 추가
  5. 기본 경로는 `data/food-products.json`
  6. 테스트에서는 `data/food-products_only10.json`을 명시해서 사용

권장:
- 파일 교체/복구 유틸을 만들지 않는다.
- 테스트는 명시적 path 기반으로 데이터 파일을 선택한다.

## 9. 결론
- retention 정책만으로는 rollback 준비까지만 된다.
- 실제 운영 복구를 위해서는:
  - 과거 인덱스 목록 조회
  - 선택 인덱스로 alias restore
  두 기능이 추가로 필요하다.
