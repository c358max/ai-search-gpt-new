## 1. 현재 프로젝트 기준 전제
- Elasticsearch는 ECK 기반으로 구동한다.
- Nori 플러그인은 커스텀 ES 이미지에 내장하는 방식이다.
- 색인/재색인/alias 전환 로직은 애플리케이션에 이미 구현되어 있다.
- 현재 샘플 데이터 기반이지만, 상용 이관 시에는 실제 상품 원천 데이터와 운영 환경 설정으로 전환해야 한다.

## 3. 권장 상용 방향

### 3.1 Elasticsearch 이미지
- 로컬에서 사용 중인 `ai-search-es:8.13.4-nori` 이미지를 원격 레지스트리로 push한다.
- 관련 스크립트:
  - `sh_bin/00_3_build_elasticsearch_nori_image.sh`
  - `sh_bin/00_4_push_elasticsearch_nori_image.sh`

권장:
- 상용에서는 로컬 이미지 사용이 아니라 **레지스트리 기반 배포**로 통일한다.
- 이미지 tag는 버전 고정:
  - 예: `registry.example.com/ai-search-es:8.13.4-nori-v1`

### 3.2 사용자 정의 사전 저장소
- 상용에서는 `ConfigMap + subPath`보다 **공유 스토리지(NAS/NFS/CSI RWX)** 방식이 더 적합하다.
- 이유:
  - 사전 파일 업데이트 경로가 명확함
  - reload 검증 구조와 연결 가능
  - mall 분리 시 디렉터리 단위 관리가 쉬움

권장 예:
- `/mnt/search-dict/user_dict_ko.txt`
- Kubernetes에서는 PVC로 연결하고 ES Pod 내부에서:
  - `/usr/share/elasticsearch/config/userdict/user_dict_ko.txt`
  로 마운트

### 3.3 Elasticsearch 데이터 저장소
- ES 데이터 디렉터리는 사용자 사전 디렉터리와 분리해야 한다.
- 데이터 저장소는 일반적으로 블록 스토리지 기반 PVC를 사용한다.
- 사용자 사전은 공유 스토리지, ES 데이터는 고성능 디스크로 분리하는 편이 안전하다.
