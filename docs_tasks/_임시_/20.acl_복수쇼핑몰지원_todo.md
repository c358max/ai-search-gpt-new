# 20. ACL 기반 복수 쇼핑몰 지원 TODO

## 0. 문서 목적
- 본 프로젝트에서 **서로 다른 2개 쇼핑몰(DB/상품 스키마 상이)**을 하나의 검색 엔진 코어로 지원하기 위한 설계/구현 TODO를 정의한다.
- 현재는 초안이며, 이후 **색인(Indexing)부터 헥사고날 아키텍처 I/F 상세**를 단계적으로 업데이트한다.

## 1. 배경 및 문제
- 쇼핑몰 A/B는 데이터 소스(DB), 컬럼 구조, 상품 속성 체계가 다르다.
- 현재 구현은 단일 쇼핑몰 전제를 기반으로 구성되어 있다.
  - `IndexingRunner` -> `IndexRolloutService` -> `ProductIndexingService` 흐름이 단일 source 데이터 기준이다.
  - `application.yml`의 `index-name`, `read-alias`, `synonyms-set`도 단일 값 기준이다.
- 현재 구조에서 쇼핑몰별 조건 분기를 코어 검색 로직에 직접 넣으면:
  - 도메인 오염
  - 분기 증가로 유지보수 난이도 상승
  - 신규 쇼핑몰 추가 시 변경 범위 확대

## 2. 기본 방향
- **코어(ES/검색 엔진)는 공통 Standard 스키마만 사용**한다.
- 외부(쇼핑몰별 DB/응답 DTO)와의 차이는 **ACL(Anti-Corruption Layer)**에서 흡수한다.

### 2.1 ACL 정의
- ACL(Anti-Corruption Layer): 외부 시스템의 모델/규칙이 내부 도메인을 오염시키지 않도록 변환/격리하는 계층

## 3. 제안 아키텍처(초안)

### 3.1 Core(공통)
- `StandardProductDocument` (색인 표준 모델)
- `StandardSearchRequest` / `StandardSearchResult` (검색 표준 모델)
- ES 인덱싱/검색 로직은 위 Standard 모델만 의존
- 단, 현재 코드에는 이미 `ProductSearchRequest`, `SearchPageResult`가 존재하므로
  - 신규 Standard 타입을 별도로 도입할지
  - 기존 타입을 공통 표준 모델로 승격할지
  를 함께 검토해야 한다.

### 3.2 Inbound ACL (색인 입력)
- Mall A/B 원천 상품 모델 -> StandardProductDocument 변환
- 역할:
  - 필드 매핑
  - 값 정규화(단위/포맷)
  - 카테고리 표준화
  - 필수값 검증/보정

### 3.3 Outbound ACL (검색 응답)
- StandardSearchResult -> Mall A/B API 응답 DTO 변환
- 역할:
  - 쇼핑몰별 응답 필드/포맷 맞춤
  - 노출 정책(배지/가격표기/옵션표기) 반영

### 3.4 멀티 쇼핑몰 운영 단위
- read alias, 동의어 세트, 사용자 사전, 카테고리 부스팅 룰을 mall 단위로 분리
  - 예: `mall-a-products-read`, `mall-b-products-read`
- 단순히 `MallType`만 추가하는 것으로는 부족하며, 실제 실행 시 아래 mall별 운영 컨텍스트가 함께 전달되어야 한다.
  - `indexName`
  - `readAlias`
  - `synonymsSet`
  - 사용자 사전 경로
  - 필요 시 카테고리 부스팅/운영 설정

## 4. 헥사고날 관점 설계 원칙
- Port는 코어가 정의, Adapter는 쇼핑몰별 구현
- SRP: 매핑/조회/응답 책임 분리
- OCP: 쇼핑몰 추가 시 기존 코어 수정 최소화(새 Adapter 추가 중심)
