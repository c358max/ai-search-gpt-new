# 12_1. 사용자정의사전 reload 수동검증 TODO

## 0. 문서 목적
- 현재 `ConfigMap + subPath` 방식을 `PersistentVolume/PVC` 기반 외부 사전 파일 마운트 구조로 바꾸고, **수동 검증을 완료하기 위한 문서**다.
- 이 문서에서는 자동화 스크립트 수정이나 운영 fallback 고도화는 다루지 않는다.

## 1. 현재 한계
- 기존에는 `ConfigMap + subPath` 방식이었다.
- 이 방식은 Pod 내부 파일이 즉시 갱신되는지 검증하기 어렵고, 현재 수동 검증 목표와 맞지 않았다.
- 현재는 외부 파일 기반 `PersistentVolume/PVC` 마운트 구조로 전환 완료했다.

## 2. 목표
1. 로컬 디렉터리를 준비한다.
2. Kubernetes `PV/PVC`로 ES Pod에 마운트한다.
3. `index-mapping.json`이 새 경로를 보도록 변경한다.
4. `with-yalpi` / `without-yalpi` 사전을 수동 교체한다.
5. ES 재시작 없이 신규 인덱스 생성 + 재색인으로 검색 결과 차이를 검증한다.

## 3. 대표 검증 시나리오
- 현재 `user_dict_ko.txt`에는 `얇은피`, `얄피`가 있다.
- 현재 `synonyms_ko.txt`에는 `만두, 교자, 얇은피, 얄피`가 있다.
- 이유:
  - 현재 운영 동의어 파일이 이미 `만두, 교자, 얇은피, 얄피`를 함께 포함하고 있다.
  - 따라서 수동 검증 문서도 실제 운영 파일 기준으로 맞춰야 한다.
  - 이번 검증의 초점은 “동의어 파일을 바꾸는 것”이 아니라 “사용자 사전 변경 시 analyzer 결과가 어떻게 달라지는지”를 확인하는 것이다.
- baseline:
  - `with-yalpi` 상태에서 `얄피`가 단일 토큰으로 유지되어야 한다.
  - analyzer 결과에 `얄피`, `만두`, `교자`, `얇은피`가 함께 보여야 한다.
- 변경 후:
  - 사용자 사전에서 `얇은피`, `얄피`를 제거한 파일로 교체
  - ES 재시작 없이 신규 인덱스 생성 + 재색인
  - 이후 `얄피`가 `얄`, `피`로 분해되는지 확인
  - analyzer 결과에서 `얄`, `피`, `만두`, `교자`가 나타나고 `얄피`, `얇은피`는 나타나지 않는지 확인

### 4.2 샘플 사전 파일
1. 정상 버전
- `.local-nas/userdict/user_dict_ko.with-yalpi.txt`

또는

주의:

- 즉, 이번 검증은 동의어 파일을 수정하지 않고 사용자 사전만 바꿔서 결과 차이를 확인한다.
- `config/userdict` 마운트 디렉터리 아래 파일명은 ASCII로 유지한다.
  - 한글 파일명이 들어간 추가 샘플 파일은 ES security/file watcher 초기화 실패를 유발할 수 있다.

## 5. YAML 변경 방향

### 5.1 `sh_bin/es-cluster-custom-image.yaml`
- 변경 완료:
  - `PersistentVolume`
  - `PersistentVolumeClaim`
  - ES Pod 디렉터리 마운트
  - 권장 mountPath:
    - `/usr/share/elasticsearch/config/userdict`

예시:
```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: ai-search-userdict-pv
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  hostPath:
    path: /Users/justinpark/idea/ai_search/ai-search-gpt/.local-nas/userdict
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: ai-search-userdict-pvc
  namespace: ai-search
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 1Gi
  volumeName: ai-search-userdict-pv
---
apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: ai-search-es
  namespace: ai-search
spec:
  nodeSets:
    - name: default
      podTemplate:
        spec:
          volumes:
            - name: userdict-volume
              persistentVolumeClaim:
                claimName: ai-search-userdict-pvc
          containers:
            - name: elasticsearch
              volumeMounts:
                - name: userdict-volume
                  mountPath: /usr/share/elasticsearch/config/userdict
```

## 6. 매핑 변경 방향

### 6.1 `src/main/resources/es/index-mapping.json`
- 변경 완료:
```json
"user_dictionary": "userdict/user_dict_ko.txt"
```

설명:
- ES config 기준 상대 경로다.
- 실제 Pod 내부 경로는:
  - `/usr/share/elasticsearch/config/userdict/user_dict_ko.txt`

왜 상대경로가 Pod 내부 `config`를 가리키는가:
- `user_dictionary` 값은 Spring Boot 프로젝트 경로가 아니라, Elasticsearch가 자신의 analyzer 설정을 해석할 때 사용하는 파일 경로다.
- 따라서 `userdict/user_dict_ko.txt`는 Elasticsearch 노드의 `config` 디렉터리 기준 상대경로로 해석된다.
- 현재 ECK/도커 환경에서 그 기준 경로는 `/usr/share/elasticsearch/config` 이다.
- 그래서 아래 두 설정이 연결된다.
  - `index-mapping.json`: `"user_dictionary": "userdict/user_dict_ko.txt"`
  - `es-cluster-custom-image.yaml`: `mountPath: /usr/share/elasticsearch/config/userdict`
- 결과적으로 Elasticsearch는 최종적으로 아래 실제 파일을 읽는다.
  - `/usr/share/elasticsearch/config/userdict/user_dict_ko.txt`

## 12. 기존 버전 Fork 개발자 적용 가이드

기존 ConfigMap 기반 버전을 fork한 개발자가 이번 작업 내용을 가져가려면, 아래 변경만 우선 반영하면 동일한 수동 검증 흐름을 만들 수 있다.

### 12.1 반드시 가져갈 코드
| 파일 | 설명 |
|---|---|
| `sh_bin/es-cluster-custom-image.yaml` | ConfigMap 마운트를 제거하고 `PersistentVolume`, `PersistentVolumeClaim`, `mountPath=/usr/share/elasticsearch/config/userdict` 구성을 가져간다. |
| `src/main/resources/es/index-mapping.json` | `user_dictionary` 경로를 `userdict/user_dict_ko.txt`로 사용하는 설정을 가져간다. |
| `sh_bin/05_1_reloadUserDictionary.sh` | ConfigMap 반영/Pod 재기동 로직을 제거하고, active 사전 복사 + Pod 내부 마운트 확인 + 재색인 흐름으로 바뀐 버전을 가져간다. |
| `sh_bin/05_2_verifyUserDictionary.sh` | 공통 port-forward 로직 사용, 기대 토큰/비기대 토큰 검증 로직이 들어간 버전을 가져간다. |
| `sh_bin/05_3_test_with_yalpi.sh` | `with-yalpi` 시나리오를 파라미터 없이 실행하는 wrapper 스크립트다. |
| `sh_bin/05_4_test_without_yalpi.sh` | `without-yalpi` 시나리오를 파라미터 없이 실행하는 wrapper 스크립트다. |
| `sh_bin/common/port_forward.sh` | ES port-forward 중복을 줄이고, 이미 포트가 열려 있으면 재사용하는 공통 로직이다. |
